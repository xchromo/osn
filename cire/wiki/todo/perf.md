- [x] **IB-P-I2** ŌĆö Stream the served ORIGINAL image instead of buffering. The no-transform serve path (no Images binding, or an account without the product) now uses `fetchAssetStream` to pipe R2's `obj.body` `ReadableStream` straight into the `Response`, so a Ōēż5 MB image is never fully materialised in Worker memory; `response.clone()` tees the stream so the Cache-API `put` and the returned body each get a copy. Existing serve-route tests (assert served bytes == original) + new `fetchAssetStream` unit tests cover it. (cire-hotpath-perf branch)

- [x] ~~PBKDF2 100k iterations + dummy-hash-on-miss ŌĆö consider lowering to 25-50k~~ ŌĆö **Stale/aspirational; closed (2026-06-21).** cire never shipped PBKDF2 claim-code hashing ŌĆö there is no iteration constant to lower. The guest claim code is stored as a **plaintext** unique `families.public_id` and matched by exact equality (`claimService.lookup` ŌåÆ `eq(families.publicId, code)`); the only hashing in cire is SHA-256 of the `cire_session` token + recovery codes (no iterations, no dummy-hash path). Brute-forcing a code is defended by the **per-IP rate limiter + Turnstile + secure-tier `SURNAME-WORD-HASH` entropy (~58-bit)**, not by hashing cost. Plaintext is acceptable for this threat model ŌĆö the code is a low-sensitivity capability printed on the paper invite, designed to be shared, not a password. (Hashing claim codes at rest, if ever wanted, is a separate security change to scope deliberately ŌĆö not a perf tweak.) See `[[wiki/systems/cire-auth]]` (root).
