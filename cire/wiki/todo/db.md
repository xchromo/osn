- [x] **`guests.nickname` column** (migration `0022_guest_nickname.sql`, `feat/cire-individual-greeting`) ‚Äî one nullable `text` column added in place via `ALTER TABLE guests ADD nickname text`. Powers the personalised single-guest greeting: a claim code covering exactly ONE guest is greeted individually ("Dear {nickname-or-firstName}") instead of "The {familyName} Family"; multi-guest codes ignore it. Sourced from the optional "Guest Nickname" column in the organiser Guests CSV; NULL ‚áí fall back to `first_name`, so every existing guest greets exactly as before. Additive + self-backfilling. LOCKSTEP mirrors updated in `cire/api/src/db/setup.ts` + `db/schema.test.ts`. Hand-written migration in `migrations_dir` (wrangler applies directly; drizzle `_journal.json` stale past 0008), so 0022 auto-applies at deploy. See `[[web]]` (greeting) + `[[spreadsheet-import]]`.
- [x] **Per-image crop columns** (migration `0021_image_crop.sql`, `feat/cire-image-crop`) ‚Äî three nullable `text` columns storing a JSON-encoded normalised crop rectangle `{x,y,w,h}` in SOURCE FRACTIONS (0..1): `wedding_invite_customisations.hero_image_crop` + `story_image_crop` and `events.event_image_crop`. NULL ‚áí the default centre `object-cover`, so every existing image renders exactly as before until re-cropped. NO source-dimension columns are needed ‚Äî the render is CSS-side (background-image fraction technique), so the stored bytes are untouched. Additive + self-backfilling (every existing row defaults NULL). Verified: all 21 migrations apply to a fresh local D1 and the table shapes match `schema.ts`; LOCKSTEP mirror updated in `cire/api/src/db/setup.ts` (the `db/schema.test.ts` mini-mirror doesn't include these two tables, so it's unchanged). The hand-written migrations sit in `migrations_dir` which `wrangler d1 migrations apply` reads directly (the drizzle `_journal.json` is stale past 0008 and not consulted), so 0021 auto-applies at deploy. See `[[api]]` + `[[web]]` + `[[security]]`.
- [x] **`families.first_opened_at` column** (migration `0020_family_first_opened.sql`, `feat/cire-invite-opened-status`) ‚Äî records the FIRST time a guest opened a family's invite with its CURRENT code (set by the guest claim path, `cire/api` `claimService.lookup`). - [x] **`events.event_image_key` column** (migration `0019_event_image.sql`, `feat/cire-event-images`) ‚Äî ONE optional image per event. A single nullable `text` column added in place via `ALTER TABLE events ADD event_image_key text`; stores the R2 object **key** (not a URL), exactly like `wedding_invite_customisations.hero_image_key`. NULL ‚áí the event renders text-only at every breakpoint (no empty image half). Additive + self-backfilling (every existing row defaults NULL ‚Äî live events have no image until one is uploaded through the organiser portal). Events have NO `updated_at`, so the served image's cache version is derived **server-side** from the key (FNV-1a digest in `versionFromKey`): a re-upload mints a fresh uuid-suffixed key ‚áí a fresh version ‚áí never served stale, and the client `?v=` is never trusted for keying (S-M1). Verified: all 19 migrations apply to a fresh local D1 and the table shape matches `schema.ts`; LOCKSTEP mirror updated in `cire/api/src/db/setup.ts`. See `[[api]]` + `[[web]]` + `[[invite-builder]]`.
