# cire/api

Backend feature work. The Elysia + Effect + Drizzle layer in `cire/api`.

- [x] **`pin.it` → canonical board URL resolution at import apply time** (`feat/cire-pinterest-resolve`) — fixes the recurring "Pinterest board embed doesn't load": the guest board widget needs a full `pinterest.com/<user>/<board>` URL but organisers paste `pin.it/...` short links, which the embed correctly refuses (see `[[web]]`). Resolution now happens **server-side, once, at import apply time** so guests get a working embed without any per-render network call. New `services/pinterest-resolve.ts` splits a **pure, unit-tested** `canonicalizePinterestBoardUrl` (resolved location string → canonical `https://www.pinterest.com/<user>/<board>/`, all query/tracking params + fragment stripped, host normalised to `www.`; single pins `/pin/<id>`, profiles, foreign hosts, and a still-`pin.it` final URL all rejected → `null`) from the network `resolvePinUrl`. `applyImport` (`services/import.ts`) resolves every created/updated event's `pinterestUrl` (concurrency-bounded, 4 at a time) into a per-event-id map before building the write set, and persists the resolved URL into `events.pinterest_url`. **SSRF guards** (outbound fetch from a prod Worker): only `pin.it` / `www.pin.it` inputs are ever fetched (exact-match allowlist — `pin.it.evil.com` / `evilpin.it` rejected), only a `pinterest.com`-family final host is accepted, `redirect: "manual"` loop capped at ≤5 hops, per-attempt `AbortController` timeout (~4s), and **any** error / timeout / non-board result falls back to the **original** URL unchanged — resolution never throws or blocks the import (the guest link-out still works either way). Unit tests in `services/pinterest-resolve.test.ts` (board pass-through, tracking-param stripping, `pin.it`-as-final rejected, non-pinterest host rejected, `/pin/<id>` single-pin kept-original, redirect-following + depth cap + fetch-error fallback, SSRF allowlist no-fetch). ⚠️ **Deploy-time / ops:** resolution runs at import time, so the **LIVE wedding's already-imported events keep their original `pin.it` URLs** until re-imported — re-run the organiser spreadsheet import (or re-resolve the stored `pinterest_url`s) to apply canonical board URLs and light up the embeds. See `[[web]]` + `[[spreadsheet-import]]`.
- [x] **Event Location is now a hard-required field** (`feat/invite-conditional-segments`) — every event must have a **Location** and a **Start** (Start was already enforced; Location is the new requirement). The invite's "Where" section + Open-in-Maps both derive from location, so an event with no place is unusable. `parseEventsCsv` (`services/spreadsheet.ts`) adds `Location` to `REQUIRED_EVENT_COLUMNS` (header missing ⇒ `MissingRequiredColumn`) and rejects any data row with an empty/whitespace-only Location cell ⇒ `MalformedSpreadsheet` with reason `"Location is required"` + 1-indexed row/column. The organiser template mirror (`cire/organiser/src/lib/import-templates.ts`) moves Location into `EVENT_REQUIRED_HEADERS` (gold "required" chip in the import help), and `ImportPanel.tsx` surfaces the specific parse error rather than a generic failure. New `services/spreadsheet.test.ts` cases: missing Location column rejected, empty + whitespace-only Location cell rejected, empty Start cell rejected, valid row with a Location passes. See `[[invite-builder]]` (Required event fields) + `[[spreadsheet-import]]`.

- [x] **RSVP CSV export** (`feat/rsvp-csv-export`) — `GET /api/organiser/weddings/:weddingId/rsvps.csv` returns a server-built `text/csv` download (`Content-Disposition: attachment`, `Cache-Control: no-store` — it carries guest PII; `X-Content-Type-Options: nosniff`). Gated by **`weddingMember()`** (owner **or** co-host — same gate as the `/guests` + `/events` dashboard reads), so a co-host can export. New `rsvpExportService.build(weddingId)` (`services/rsvp-export.ts`) builds **one row per guest** — Family Code (`families.public_id`), Family Name, Guest First/Last Name, **one column per event** (named by event name, ordered by event `start_at` parsed as epoch, then `sort_order`/id), then **Dietary Requirements**. Per-event cell distinguishes four states: **blank** = not invited (no `guest_events` row), **`No response`** = invited but no RSVP, **`Attending`** / **`Not attending`** / **`Maybe`** map the `rsvps.status` enum (`attending`/`declined`/`maybe`). **Guests who haven't RSVP'd at all are still included** (cells `No response` for invited events, blank otherwise). Host-kind families excluded via the same `ne(families.kind, "host")` filter as `getAllGuests`. **Rows sorted alphabetically by Family Code.** CSV serialiser (`toCsv` + `sanitiseCsvCell`) applies the same `= + - @` formula-marker set as the import parser (`spreadsheet.ts`) — but the export **neutralises** (prefixes `'`) rather than rejects, since the data is guest-supplied and we still want to surface it; RFC 4180 quoting + CRLF line endings. `weddingsService.slugOf` added for the `cire-rsvps-<slug>.csv` filename. Frontend: a "Download RSVPs (CSV)" button in `GuestTable.tsx` (`authFetch` → blob download via the extracted shared `lib/download.ts` `downloadBlob`/`downloadCsv` helper, lifted out of `ImportPanel.tsx`). See `[[wiki/systems/cire-auth]]` (root, capability matrix).
- [x] **`GET /api/primary-wedding` + preview-code returns slug** (`feat/guest-ssr-path-routing`) — supports the guest site's SSR path-routing. New **public** `GET /api/primary-wedding` (`routes/primary-wedding.ts`, no auth, `cache-control: no-store`) returns `{ slug }` for the deployment's primary wedding (the sole one, or the **most-recently-created** when several exist via `weddingsService.primaryWeddingSlug()`), **404** when none — the guest bare domain (`/`) redirects to `/<slug>` with no build-time slug var. `hostCodeService.ensureForWedding` + `POST /api/organiser/weddings/:weddingId/preview-code` now return `{ publicId, slug }` so the organiser preview opens `${CIRE_WEB_URL}/<slug>?code=…`. Invite route 404 behaviour (`GET /api/invite/:slug` → 404 on unknown slug) unchanged. See `[[invite-builder]]` (Guest rendering).
- [x] **Co-hosts can use spreadsheet import + the invite builder** — the import routes (`/import/{preview,apply,revert,list}`) and the invite-builder read + customisation routes (`GET /invite`, `PUT /invite/text`, `PUT /invite/theme`, `POST|DELETE /invite/image/:slot`) moved from `weddingOwner()` to `weddingMember()`, so a co-host gets the **full** wedding-content surface — they preview AND apply imports and edit the invite, not just read. Owner-only stays narrow: re-mint / regenerate / mark-shared / preview-code, the co-host write routes (`POST/DELETE /hosts`), and wedding deletion. `weddingMember()` fails closed if the host/ARC lookup is unavailable. - [x] **Co-host handle autocomplete** — the follow-up to the handle-display PR below. New `GET /api/organiser/handle-search?q=<prefix>` route, gated by `osnAuth()` **only** (any signed-in organiser; NOT wedding-scoped — it's not tied to a `:weddingId`), with a light per-IP rate limit (60/min). Proxies a **new** osn-api `GET /graph/internal/profile-search?prefix=&limit=` (`graph:read`) via `osn-bridge.ts` `createArcHandleSearchResolver` / `createHandleSearchResolverFromEnv`, injected through `createApp({ resolveOsnHandleSearch })` (same env wiring + key-optional guard as the display/handle/account resolvers). **Key-optional + fail-soft**: absent/malformed ARC key or unreachable osn-api ⇒ `{ profiles: [] }`, never a 503/500; the manual add-by-handle path is unaffected. **Enumeration guardrails live in osn-api**: min prefix length 2 (short/empty ⇒ empty list, not error), tombstoned accounts excluded, ordered + hard-capped at 10 (default 8), backed by a new `users_handle_idx` B-tree index on `osn-db` `users.handle` (migration `0001`). `HostsPanel.tsx` upgraded to a debounced, keyboard-navigable combobox. ⚠️ **Deploy-time**: the osn-db index migration is **manual** (not in CI) + osn-api must redeploy to serve the new endpoint — see root `[[wiki/runbooks/production-deploy]]` §4.3. **Privacy**: handles are public @-identifiers; organiser-gated, min-length 2, ≤10 results — same class as social-app @-mention autocomplete. See `[[wiki/systems/cire-auth]]` (root).
- [x] **Host list shows handle, not profileId** — `GET /api/organiser/weddings/:weddingId/hosts` previously returned only `osnProfileId`, so after a reload the portal showed the raw `usr_*` id (the handle survived only on the POST add-response). `wedding_hosts` stores **only** the profile id (never the handle — no denormalisation), so the list now resolves profileId→handle **live**: `osn-bridge.ts` `createArcProfileDisplayResolver` / `createProfileDisplayResolverFromEnv` batch the row ids into one ARC-gated (`graph:read`) `POST /graph/internal/profile-displays` call and the route merges `{handle, displayName}` into each host (profileId stays as the fallback).
