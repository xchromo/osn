# cire/api

Backend feature work. The Elysia + Effect + Drizzle layer in `cire/api`.

- [x] **First-party CSP violation-report collector ‚Äî `POST /api/csp-report`** (`feat/cire-csp-report-collector`) ‚Äî a PUBLIC, unauthenticated collector (`routes/csp-report.ts`) so the cire/web Report-Only CSP (#205, see `[[security]]`) surfaces what would break in real guests' browsers, logged to observability with NO third-party service. Normalises BOTH wire formats ‚Äî legacy `report-uri` (`application/csp-report`, `{ "csp-report": { ‚Ä¶ } }`) and the Reporting API (`application/reports+json`, an array of `{ type: "csp-violation", body: { ‚Ä¶ } }`) ‚Äî by sniffing the body shape, and tolerates malformed input. Each violation is logged via `Effect.logWarning` (through `runCire`) with a small **bounded** slice: the effective/violated directive, the blocked URI **reduced to its origin** (`reduceBlockedUri`, or truncated to 128 chars ‚Äî never the full URL with its query, which could carry a claim code), the document **path only** (`reduceDocumentPath`, query/hash stripped ‚Äî the slug is public, not PII), and the disposition. A bounded-cardinality `cire.csp.report` counter is incremented by a fixed `effectiveDirective` label (`bucketCspDirective` ‚Üí `other` for anything unknown; the blocked URI is NEVER a metric attribute). **Always answers 204 No Content**, reflecting nothing. **Abuse-hardened** (public + creds-less): 16 KB body cap (declared `Content-Length` pre-check + post-read guard), a generous per-IP rate limit (`cspReportLimiter`, ~60/min, reusing `middleware/rate-limit`) that **fail-OPENs** to a silent drop (still 204, never 429/500 on a fire-and-forget endpoint), and **no D1 write** (log + metric only ‚Äî no write-amplification DoS). No Turnstile, no Origin/auth gate (browsers POST reports automatically with no creds and a cross-origin/null Origin) ‚Äî mounted in `createApp` **before** the CSRF origin guard so it isn't 403'd. The cire/web directives (`report-uri` + `report-to csp-endpoint`) + the `Reporting-Endpoints` header are the `[[web]]`/`[[security]]` side. **View reports:** `wrangler tail cire-api` (Workers tail) or the Cloudflare observability dashboard ‚Äî still visible in DevTools too. No DB migration. New tests: legacy + Reporting-API bodies ‚Üí 204 each entry handled, malformed/oversize ‚Üí 204 no crash, blocked-uri reduced to origin / truncated, document query stripped, cross-origin reachability, rate-limit drops to 204. See `[[security]]` + `[[web]]`.
- [x] **Crop distortion fix + dims capture + aspect presets** (`feat/cire-crop-aspect-ratios`) ‚Äî fixes a guest-side crop **distortion** bug and lets the organiser pick an aspect ratio. **Root cause** (frontend): the dims-free CSS render used a TWO-value `background-size` (`Wx% Wy%`), scaling width/height INDEPENDENTLY, so any crop whose aspect differed from the display box stretched. **Fix**: a SINGLE-value `background-size` (uniform scale) + the box adopting the crop's true pixel aspect ‚Äî see `[[web]]`. **API change (no migration)**: computing that pixel aspect needs the source image's natural dimensions, so `ImageCrop` in `schemas/invite.ts` gains **optional** `natW`/`natH`. The crop columns are plain JSON `TEXT`, so the JSON shape is simply **widened** ‚Äî **no DB migration** (`[[db]]` untouched). `isValidCrop` / the `ImageCropBody` Effect schema / `decodeCrop` now accept + validate the dims **only when present** (a present-but-bad dim ‚Äî 0/negative/NaN/‚àû/non-number ‚Äî is rejected so a downstream aspect computation can't divide by zero or go non-finite) and **tolerate their absence** (a legacy `{x,y,w,h}` crop saved before this field stays valid + decodes unchanged). New pure `cropAspect(crop)` helper (`(w¬∑natW)/(h¬∑natH)`, or `null` without dims). The same optional dims are threaded onto the claim `EventSummary` schema (`schemas/claim.ts`). The dims round-trip through `PUT .../image/:slot/crop` save ‚Üí the public invite (`getForSlug`) + claim payloads. The bounds gate (each value 0..1, w/h > 0, x+w ‚â§ 1, y+h ‚â§ 1) is unchanged ‚Äî only `x,y,w,h` reach the guest-facing inline style, so the dims are validated but NOT part of the security gate. New tests: dims accepted/round-tripped on save + read, legacy crop (no dims) still valid, present-but-bad dim rejected (400 on the body), `cropAspect` maths. **Aspect presets** (Original / 16:9 / 3:2 / 4:3 / 1:1 / 4:5 / Free, per-slot Original defaults hero 16:9 / story 3:2 / event 4:3) live in the organiser editor ‚Äî see `[[web]]` / `ImageCropModal`. - [x] **Reliable "invite Opened" status on a real family-code claim** (`feat/cire-invite-opened-status`) ‚Äî replaces reliance on the false-positive-prone "Sent" status (which only meant the organiser clicked "Copy message") with an "Opened" status driven by an ACTUAL guest claim. `claimService.lookup` records the first open on a successful claim. The write is **idempotent** ‚Äî a guarded `UPDATE families SET first_opened_at = ? WHERE id = ? New tests: claim sets `first_opened_at` on first guest claim; a SECOND claim does not overwrite it (idempotent); a host-preview claim does NOT set it; a claim write failure does not fail the claim (best-effort); `getAllGuests` + the `/guests` route surface `firstOpenedAt`; remint clears it.
