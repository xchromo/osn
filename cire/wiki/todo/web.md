- [x] **Crop editor still dead in production — root cause was CORS cache poisoning, not geometry** (`claude/cire-image-crop-upload-7yk9c7`) — after the #240 geometry/containment fixes the editor still opened dead on the deployed portal. Root cause (proven in headless Chromium against real cropperjs 2.1.1): the dashboard thumbnail loads the invite image URL as a plain no-cors `<img>` first, the API responds `Cache-Control: immutable` + `Vary: Accept` (no `Vary: Origin`), so the browser caches the response **without** CORS headers; `ImageCropModal`'s `<img crossOrigin="anonymous">` for the identical URL was then answered from that cache entry and hard-failed the CORS check without touching the network — cropperjs `$ready` rejected and the selection never seeded. Fix: dropped `crossOrigin` from the modal image (the editor reads only geometry + `naturalWidth/Height`, never canvas pixels, so a non-CORS image is sufficient); the no-crossorigin constraint + the `Vary: Origin` prerequisite for any future pixel-access feature are documented in `[[invite-builder]]`. `crossOrigin` had been on the modal since the original editor (#200) — the feature likely never worked on `cireweddings.com`; #215/#240 fixed real but secondary geometry bugs.
