- [x] **Crop editor still dead in production ‚Äî root cause was CORS cache poisoning, not geometry** (`claude/cire-image-crop-upload-7yk9c7`) ‚Äî after the #240 geometry/containment fixes the editor still opened dead on the deployed portal. Root cause (proven in headless Chromium against real cropperjs 2.1.1): the dashboard thumbnail loads the invite image URL as a plain no-cors `<img>` first, the API responds `Cache-Control: immutable` + `Vary: Accept` (no `Vary: Origin`), so the browser caches the response **without** CORS headers; `ImageCropModal`'s `<img crossOrigin="anonymous">` for the identical URL was then answered from that cache entry and hard-failed the CORS check without touching the network ‚Äî cropperjs `$ready` rejected and the selection never seeded. Fix: dropped `crossOrigin` from the modal image (the editor reads only geometry + `naturalWidth/Height`, never canvas pixels, so a non-CORS image is sufficient); the no-crossorigin constraint + the `Vary: Origin` prerequisite for any future pixel-access feature are documented in `[[invite-builder]]`. `crossOrigin` had been on the modal since the original editor (#200) ‚Äî the feature likely never worked on `cireweddings.com`; #215/#240 fixed real but secondary geometry bugs.
- [x] **Organiser dashboard: guest-roster + event-list CSV download buttons** ‚Äî the Guests tab gains a "Download guests (CSV)" button beside the existing RSVPs one (`GuestTable.tsx` ‚Äî the two share one generalised `exportCsv(kind)` handler + in-flight guard), and the Events tab gains a "Download events (CSV)" button in its `SectionIntro` actions slot (`EventTable.tsx`, which now takes a `weddingSlug` prop threaded from `DashboardTabs.tsx` for the `cire-events-<slug>.csv` filename). Both hit the new server-built, `weddingMember()`-gated `GET ‚Ä¶/guests.csv` / `‚Ä¶/events.csv` endpoints (`[[api]]`) via `authFetch` and hand the Blob to the shared `lib/download.ts` helper ‚Äî same 401-redirect + toast pattern as the RSVP export. Component tests cover endpoint + slug-based filename + error toast for both buttons.
- [ ] **HEIC image uploads ‚Äî convert client-side before upload (cire/organiser)** ‚Äî the invite/event image upload accepts only JPEG/PNG/WebP by magic-byte sniff (`detectImageType`, deliberate server posture), so iPhone photos in HEIC are rejected with a 415 that reads like a bug to organisers. Decision (2026-07-04): keep the server allow-list as-is and **transcode HEIC ‚Üí JPEG/WebP in the browser** before `POST .../image/:slot` ‚Äî e.g. lazy-load a `heic2any`/libheif-wasm decoder only when a `.heic`/`.heif` file is picked (keep it out of the dashboard chunk, same concern as CB-P-I2 in `[[perf]]`), then hand the converted blob to the existing upload path. Also widen the file input's `accept` so HEIC is pickable, and show a "converting‚Ä¶" state (wasm decode of a 12MP photo is not instant). Surfaced while investigating the crop-editor regression (PR #240).
- [ ] **Pinterest moodboard embed not rendering in practice ‚Äî needs live DevTools deep-dive** ‚Äî after re-enabling the embed on all devices (#220) + backfilling the 4 live events to canonical board URLs, the board still doesn't visibly render for the product owner. Ruled out (2026-06-21): URL resolution (boards resolve to canonical `pinterest.com.au/<user>/<board>/`), data freshness (events come from the live `/api/claim` response, not the image cache), CSP (Report-Only, all Pinterest origins allowlisted), and board availability ‚Äî Pinterest's pidgets API (`widgets.pinterest.com/v3/pidgets/boards/<user>/<board>/pins/`) returns `status: success` with pins. Remaining suspect: Pinterest's own client-side `pinit_main.js` widget rendering (long-standing mobile flakiness). NEXT: live DevTools on the deployed site ‚Äî does `pinit_main.js` load, does the pidgets XHR succeed, does an `<iframe>` get inserted into the embed container? Possible minor follow-up: normalise the stored board host to `www.pinterest.com` (Pinterest's canonical) rather than the regional `.com.au` (harmless for the slug-based pidgets call, just tidier). See `[[api]]` (pin.it resolution) + `PinterestBoard.tsx`.
- [x] **"Link my Pulse account" affordance (account-linking frontend)** (`feat/cire-pulse-account-link`) ‚Äî `PulseAccountLink.tsx` renders post-claim (non-preview) inside `InvitePage.tsx`, wrapped in its own `@osn/client` `AuthProvider`.
